# Telegram 下载系统项目具体教程（实操版）

本教程基于你当前环境整理：

- NAS: `192.168.3.11`
- Docker 项目目录: `/vol00/WD40PURX-64D85Y0_1/备份/telegram/tg-media-app`
- 前端端口: `13000`
- 后端端口: `18000`
- 下载根目录: `/vol00/WD40PURX-64D85Y0_1/备份/telegram`

---

## 0. 先理解这个项目的 3 个服务

`docker-compose.yml` 里有三个容器：

1. `tg-media-worker`
作用：连接 Telegram，真正执行下载。

2. `tg-media-backend`
作用：提供 API，管理数据库、文件列表、仪表盘数据。

3. `tg-media-frontend`
作用：你访问的网页管理后台。

核心原则：
- `worker` 和 `backend` 必须看到同一个 `/downloads`，否则会出现“明明在下载但仪表盘显示 0”。

## 1. 第一次部署（或重装）

### 1.1 准备目录

在 NAS 中确认目录存在：

```bash
/vol00/WD40PURX-64D85Y0_1/备份/telegram/tg-media-app
```

### 1.2 配置 `.env`

打开项目根目录 `.env`，至少要有：

```env
API_ID=你的API_ID
API_HASH=你的API_HASH
PHONE_NUMBER=你的手机号（含国家码）
SESSION_NAME=/session/feiniu_user
DOWNLOAD_DIR=/downloads
TARGET_CHATS=@wodsidnfu,@sjxbxhsks
ALLOW_EXTS=.mp4,.mkv,.mov,.avi,.jpg,.jpeg,.png,.webp
DOWNLOAD_HISTORY=true
HISTORY_LIMIT=2000
MAX_RETRIES=3
RETRY_DELAY=5
MAX_FILE_SIZE_MB=0
BACKEND_PORT=18000
FRONTEND_PORT=13000
DOWNLOADS_VOLUME=/vol00/WD40PURX-64D85Y0_1/备份/telegram
```

`DOWNLOADS_VOLUME` 必须是 NAS 的真实下载目录。

### 1.3 启动

在项目目录执行：

```bash
docker compose up -d --build
```

### 1.4 检查运行状态

```bash
docker ps --format 'table {{.Names}}\t{{.Status}}\t{{.Ports}}'
```

你应看到：
- `tg-media-frontend` -> `13000`
- `tg-media-backend` -> `18000`
- `tg-media-worker` -> running

## 3. Telegram 下载流程怎么跑

1. `worker` 启动后连接 Telegram。
2. 扫描配置频道历史消息（`DOWNLOAD_HISTORY=true`）。
3. 把可下载媒体写入等待队列。
4. 按顺序下载，写到 `/downloads/...`。
5. 同时写入数据库，前端显示状态/进度。

注意：
- 队列大时，某文件“没下载”通常是还没轮到。
- 日志里若反复出现 `Waiting for 3 seconds`，是 Telegram 限速行为，正常。

## 4. 文件管理使用教程

### 4.1 同步历史已下载文件（非常重要）

如果“文件管理看不到历史文件”，先同步：

- 前端点击“同步下载目录”
- 或调用接口：

```http
POST /api/downloads/reconcile-files
```

这会把下载目录里已有文件回填到数据库。

### 4.2 列表/视图切换

- 列表：信息密度高，适合排查。
- 视图：强调图片内容，适合预览。

### 4.3 视频封面机制

视频封面由 NAS 端 `ffmpeg` 取固定帧生成缓存，不会把整段视频下载到你电脑后再截帧。

## 5. 仪表盘使用教程

当前你最关键看这几块：

1. 当前下载详情
- 文件名
- 进度条
- 当前字节 / 总字节
- 实时速度

2. 统计区
- 总下载数、成功、失败、跳过、近24小时新增、总文件体积

3. 网络速率区
- 下载任务速率
- 下载任务进度
- 下载中任务数
- 排队任务数

这四项更适合判断“是否真的在下载 + 是否堵在队列”。

## 7. 常见问题与解决

## 7.1 文件不下载 / 卡住

先看：

```bash
docker logs --tail 200 tg-media-worker
```

若是网络抖动（`Connection lost`）或限速，通常会自动重试。

如果长期“DOWNLOADING 但不增长”：
- 检查任务是否卡死
- 可把卡死任务改回 `WAITING` 重新排队

## 7.2 仪表盘速度不对、进度条不动

99% 是挂载不一致。

检查 backend 与 worker 的 `/downloads`：

```bash
docker inspect tg-media-backend --format '{{json .Mounts}}'
docker inspect tg-media-worker --format '{{json .Mounts}}'
```

必须都指向同一个 NAS 真实目录。

## 7.3 文件管理没有内容

- 执行 `reconcile-files`
- 再确认 `/downloads` 挂载一致
- 刷新页面

## 7.4 端口冲突

改 `.env`：

```env
BACKEND_PORT=18000
FRONTEND_PORT=13000
```

然后：

```bash
docker compose up -d --build
```

## 9. 安全与备份建议

1. 不要把 `.env`、`session/*.session` 上传到公开仓库。
2. `data/app.db` 建议每天自动备份。
3. 重要下载目录建议启用 NAS 快照。

---

## 10. 你这套环境的一句话排障口诀

“先看 worker 日志，再看 active-downloads，再看 backend/worker 的 `/downloads` 是否同一挂载。”

只要这三步做了，绝大多数问题都能快速定位。
